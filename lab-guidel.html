Capture Setup & tcpdump

Capturing SSH traffic on the Azure VM and getting it into Wireshark

TCPDUMP CAPTURE METHODS FILE TRANSFER
Capture Methods Overview

You have two options for getting SSH traffic into Wireshark. Option A (recommended for Azure) is to run tcpdump on the VM, write a .pcap file, and transfer it locally. Option B is a remote live capture piped directly into Wireshark.

Option A — tcpdump on VM → SCP to Local (Recommended)

1
SSH into your Azure VM
TERMINAL
ssh azureuser@52.246.249.89
2
Start a targeted capture on port 22 — The -w flag writes raw packets to a .pcap file. The port 22 filter keeps the file manageable.
AZURE VM — CAPTURE SSH TRAFFIC FOR 5 MINUTES
# Capture all port 22 traffic for 5 minutes, write to pcap
sudo tcpdump -i eth0 port 22 -w /tmp/ssh_bruteforce.pcap -v

# With timestamp precision and snap length
sudo tcpdump -i eth0 port 22 -w /tmp/ssh_bruteforce.pcap \
  --time-stamp-precision=nano -s 0

# Run for exactly 300 seconds then stop
sudo timeout 300 tcpdump -i eth0 port 22 \
  -w /tmp/ssh_bruteforce.pcap -s 0

# Check how much was captured
ls -lh /tmp/ssh_bruteforce.pcap

SSH Protocol Deep Dive

Understanding the packet-level anatomy of SSH — what's visible, what's encrypted, and what brute force looks like

TCP/IP SSH2 PROTOCOL KEX
SSH Connection Phases

Every SSH connection — whether legitimate or an attack attempt — goes through the same phases. Understanding these phases tells you exactly which packets you're looking at in Wireshark.

Phase 1: TCP Three-Way Handshake

Before any SSH protocol exchange, TCP establishes a connection. In a brute force attack you'll see hundreds of these per minute.

ATTACKER AZURE VM :22
SYN ────────────────────────► [TCP Flags: S] seq=0 sport=random
◄──────────────────────── SYN-ACK [TCP Flags: SA] seq=0 ack=1
ACK ────────────────────────► [TCP Flags: A] Connection ESTABLISHED

Phase 2: SSH Version Exchange (PLAINTEXT — Visible!)

Immediately after the TCP handshake, both sides exchange version strings. These are sent in plaintext — your best source of attacker tool intelligence.

VM sends: SSH-2.0-OpenSSH_8.9p1 Ubuntu-3ubuntu0.6
Attacker: SSH-2.0-libssh-0.9.6 ← tool fingerprint!
Attacker: SSH-2.0-PUTTY_Release_0.79 ← or this
Attacker: SSH-2.0-Go ← or this (automated Go tool)
🔍
SOC Intelligence: Version String Fingerprinting
Common brute force tool signatures in SSH version strings: libssh (Hydra, custom scripts), SSH-2.0-Go (automated Go-based scanners), SSH-2.0-paramiko (Python-based tools, Metasploit modules), SSH-2.0-JSCH (Java SSH Client — often in enterprise scanners or automated attack tools). These strings are searchable threat intelligence.

Phase 3: Key Exchange Initialization (KEX)

Both sides negotiate encryption algorithms. The packet is visible but the payload data is algorithmic — not credentials. Wireshark labels this SSH2_MSG_KEXINIT.

WIRESHARK PACKET DETAIL — KEXINIT
SSH Protocol
  ├─ Packet Length: 1028
  ├─ Padding Length: 4
  └─ Key Exchange
       Message Code: Key Exchange Init (20)
       Cookie: a3f91bc2d4e85a71...
       KEX Algorithms: curve25519-sha256,ecdh-sha2-nistp256,diffie-hellman-group14-sha256
       Server Host Key Algorithms: rsa-sha2-512,rsa-sha2-256,ecdsa-sha2-nistp256
       Encryption c→s: chacha20-poly1305@openssh.com,aes128-ctr
       Encryption s→c: chacha20-poly1305@openssh.com,aes128-ctr
       MAC c→s: umac-64-etm@openssh.com,hmac-sha2-256-etm
       ...
💡
KEX Algorithm Fingerprinting
Legitimate clients typically support many KEX algorithms in priority order. Automated brute force tools often offer a stripped-down, unusual, or non-standard KEX list — another tool fingerprint. Compare the attacker's KEXINIT against standard OpenSSH to spot anomalies.

Phase 4: Authentication Exchange (ENCRYPTED)

After key exchange completes, all authentication is encrypted. You cannot see usernames or passwords in the packets. However, you can observe:

  • The number of SSH packets per connection (auth attempts correlate to packet count)
  • The connection duration — failed auth ends the connection quickly
  • The TCP RST or FIN pattern after each attempt (immediate disconnection = failed auth)
  • The timing delta between connections (sub-second = automated tool)

Complete Brute Force Connection Lifecycle

→ SYN TCP connect attempt
← SYN-ACK VM accepts connection
→ ACK Handshake complete
← SSH-2.0-OpenSSH_8.9p1 Ubuntu VM version string (plaintext)
→ SSH-2.0-libssh-0.9.6 Attacker version (plaintext) ← FINGERPRINT
→ KEXINIT Attacker KEX algorithms
← KEXINIT VM KEX algorithms
→ KEX DH Init DH key exchange
← KEX DH Reply + NEW_KEYS Encryption starts here
→ [ENCRYPTED] Auth Request Username + password (encrypted)
← [ENCRYPTED] Auth Failure Failed — sends disconnect
→ FIN/RST Connection closed
↺ Repeat immediately for next password attempt

Wireshark Display Filters

The exact filter syntax for SSH brute force triage — build a filter toolkit

DISPLAY FILTERS BPF SYNTAX TRIAGE
💡
Display Filters vs Capture Filters
Capture filters (tcpdump BPF syntax) are applied during capture — they determine what gets recorded. Display filters (Wireshark syntax) are applied after — they show/hide packets from an existing pcap. They use different syntax. This module covers display filters.

Tier 1 — Baseline SSH Filters

tcp.port == 22
All TCP traffic on port 22. Your starting point.
ssh
Show only packets Wireshark has decoded as SSH protocol.
tcp.dstport == 22
Inbound connections only — attacker → VM traffic.
tcp.srcport == 22
Outbound responses only — VM → attacker.

Tier 2 — TCP Flag Filters (Brute Force Patterns)

tcp.flags.syn == 1 and tcp.flags.ack == 0
SYN-only packets. Each = one connection attempt. High count = scanning/brute force.
tcp.flags.reset == 1
RST packets. After failed auth, many tools send RST to close quickly. High RST count = aggressive tool.
tcp.flags.fin == 1
FIN packets — graceful close after auth failure. Less aggressive tools use FIN over RST.
tcp.analysis.retransmission
Retransmitted packets — can indicate network issues or a slow brute forcer hitting timeouts.

Tier 3 — SSH Content Filters

ssh.message_code == 20
SSH KEXINIT packets only. One per connection attempt — useful for counting attempts.
ssh.protocol
Packets containing the SSH version string (the plaintext banner exchange).
frame contains "SSH-2.0"
Raw text search for SSH version strings. Finds banners in any direction.
frame contains "libssh"
Find packets from libssh-based tools (Hydra, custom scripts). Swap for other tool strings.

Tier 4 — IP & Time Filters

ip.src == 185.224.128.0
Filter to a specific attacker IP. Replace with IPs you've identified.
ip.src != 52.246.249.89
Exclude your own VM IP — shows all external traffic only.
frame.time_relative < 60
First 60 seconds of the capture. Useful for isolating a burst.
tcp.time_delta < 0.1
Packets arriving faster than 100ms. Flags automated tools.

Tier 5 — Power Combinations

WIRESHARK DISPLAY FILTER — COMPLEX EXPRESSIONS
# SYN packets to port 22 from non-RFC1918 IPs (external attackers)
tcp.flags.syn == 1 and tcp.flags.ack == 0 and tcp.dstport == 22 and
not (ip.src_host matches "^10\\..*|^192\\.168\\..*|^172\\.(1[6-9]|2[0-9]|3[01])\\..*")

# All SSH connection attempts with RST close (fast failed auth pattern)
tcp.dstport == 22 and (tcp.flags.syn == 1 or tcp.flags.reset == 1)

# SSH version banners only (attacker fingerprinting)
tcp.dstport == 22 and frame contains "SSH-2.0"

# Short-lived SSH connections (brute force pattern — connect, fail, close)
tcp.dstport == 22 and tcp.flags.syn == 1 and tcp.stream eq 0
🎯
SOC Triage Order
In a real investigation: start with tcp.port == 22 to scope the volume. Then tcp.flags.syn == 1 and tcp.flags.ack == 0 and tcp.dstport == 22 to count inbound connection attempts. Then frame contains "SSH-2.0" to extract tool fingerprints. Then filter by specific attacker IPs from Sentinel to correlate your two data sources.

TCP Connection Pattern Analysis

Reading the attack signature from TCP flags alone — before touching SSH content

TCP FLAGS ATTACK PATTERNS TIMING ANALYSIS
The Brute Force TCP Signature

A brute force attack has a distinctive TCP pattern. Instead of normal traffic (connect, use session for minutes, graceful close), you see: rapid SYN bursts → connection established → very short session → RST/FIN → immediate reconnect. This cycle repeats hundreds of times per minute.

Pattern 1: SYN Burst Rate

1
Apply filter: tcp.flags.syn == 1 and tcp.flags.ack == 0 and tcp.dstport == 22
2
Look at the Time column. In a brute force attack, timestamps between SYN packets will be sub-second (e.g., 0.002s apart). Legitimate logins are seconds or minutes apart.
3
Check the bottom status bar: Displayed: 842 SYN packets in 5 minutes = 168 connection attempts per minute = automated attack.

Pattern 4: RST vs. FIN Analysis

🔍
RST vs FIN — What It Tells You
TCP RST (Reset): Abrupt connection termination. The attacker isn't waiting for graceful close — they're hammering as fast as possible. Hydra, Medusa, and high-speed custom tools typically use RST. TCP FIN: Graceful close. Slower tools, or tools that try to be less noisy, send FIN. If you see only RSTs: high-speed automated attack. Mix of FIN and RST: possibly multiple tools or slower tool with some timeout handling.
WIRESHARK — ANALYZE RST vs FIN RATIO
# Filter RST packets to port 22 traffic
tcp.flags.reset == 1 and tcp.port == 22

# Filter FIN packets to port 22 traffic  
tcp.flags.fin == 1 and tcp.port == 22

# Check packet counts in status bar
# RSTs >> FINs = aggressive automated tool
# FINs ~= RSTs = mixed or slower tool

Pattern 5: Time Delta Between Connections

1
Edit → Preferences → Columns → Add a Delta time column (frame.time_delta) to see time between each packet.
2
Filter to SYN packets only. A delta of 0.001–0.010 seconds between SYNs means 100–1000 attempts/second — definitively automated.
3
Human typing speed is ~1–5 seconds between attempts. Anything under 100ms is a machine.

Constant rate (flat line): Throttled tool with delay between attempts — trying to avoid detection. Burst pattern (spikes): Intermittent scans, possibly coordinated across multiple IPs. These patterns tell you about attacker sophistication and tool configuration.

T1110 — Brute Force (Credential Access)
MITRE ATT&CK T1110 covers four sub-techniques: .001 Password Guessing, .002 Password Cracking, .003 Password Spraying, and .004 Credential Stuffing. Each has distinct network signatures visible in Wireshark.
T1110 Sub-Technique Network Signatures
Sub-techniqueNetwork PatternWireshark Filter
T1110.001
Password Guessing		 Many connections from one IP, rapid SYN rate, sequential port use ip.src == [ip] and tcp.dstport == 22 and tcp.flags.syn==1
T1110.003
Password Spraying		 Many IPs, slower rate per IP, wider IP distribution, same timing pattern across sources tcp.dstport == 22 and tcp.flags.syn==1 → check Endpoints for IP diversity
T1110.004
Credential Stuffing		 Similar to .001 but with variable timing (loading credential pairs), often paramiko/Go tools frame contains "paramiko" or frame contains "SSH-2.0-Go"

Network IOC Checklist for T1110

1
High SYN rate to :22 — >10 SYN packets/second to port 22 from single IP is a clear indicator
2
Short connection duration — Average TCP stream <2 seconds with no data transfer beyond SSH handshake
3
High RST rate — RST count approximately equal to SYN count (one failed attempt per cycle)
4
Automated tool version strings — libssh, paramiko, Go, JSCH in SSH protocol field
5
Stripped KEX algorithm list — Fewer algorithms than standard OpenSSH, especially missing modern curve25519
6
IP reputation — Source IPs in known scanner/attack infrastructure ranges (check AS numbers)

Interview Prep & SOC Context

Questions you'll face in SOC analyst interviews — with full answers grounded in this lab

INTERVIEW SOC ANALYST TECHNICAL DEPTH

Technical Packet Analysis Questions

Walk me through what an SSH brute force attack looks like at the packet level. +
Strong answer: "An SSH brute force at the packet level shows a distinctive TCP pattern. Each attempt starts with a standard three-way handshake — SYN, SYN-ACK, ACK. Then the SSH version strings are exchanged in plaintext — that's actually where you can fingerprint the attacker's tool. After the version exchange, both sides do a KEX Init to negotiate encryption algorithms, then the DH key exchange completes and encryption starts. The auth attempt itself is encrypted, but the session is extremely short — typically under 2 seconds — and ends with either a TCP RST (aggressive tool) or FIN. This cycle repeats hundreds of times per minute. The key indicator is the ratio of SYN packets to unique source ports to port 22 — in a brute force, every SYN generates its own short-lived TCP stream."
What Wireshark display filters would you apply first when triaging SSH-related traffic? +
Strong answer: "My triage sequence: First, tcp.port == 22 to scope total SSH volume. Then tcp.flags.syn == 1 and tcp.flags.ack == 0 and tcp.dstport == 22 — this counts inbound connection attempts; the packet count tells me immediately if I'm looking at a brute force. Then frame contains "SSH-2.0" to see version strings and fingerprint attacker tools — this is plaintext and actionable for threat intel. Finally, filter by specific IPs from my SIEM to correlate the two data sources. I'd also check Statistics → Conversations to see the duration distribution — brute force attempts are all under 2 seconds, legitimate sessions run much longer."
SSH traffic is encrypted. What can you actually determine from a Wireshark capture? +
Strong answer: "More than most people expect. The SSH version string exchange happens in plaintext — you can see the exact client software version string, which often fingerprints the attack tool: libssh points to Hydra or custom C tools, paramiko means Python scripts or Metasploit, SSH-2.0-Go means an automated Go scanner. The KEX Init is visible — attackers using stripped-down or non-standard algorithm lists reveal their tools. At the TCP layer, you get connection timing (sub-100ms between attempts = automated), RST vs FIN patterns (RST = aggressive), session duration distribution (brute force = all <2 seconds), and source IP data for threat intel. The only thing truly opaque is the credential content — you can't see usernames or passwords — but everything about the attack mechanics and tooling is visible."
How do you differentiate SSH brute force from legitimate failed login attempts in a pcap? +
Strong answer: "Three key differentiators. First, rate: brute force tools operate in milliseconds — I've seen 100+ connection attempts per minute. Legitimate users might try a password twice, maybe three times. Second, session duration: a brute force attempt completes the SSH handshake and disconnects in under 2 seconds. A human taking multiple tries still takes 3–10 seconds per attempt because of authentication timeout handling. Third, tool version strings: legitimate users show standard SSH client versions — OpenSSH, PuTTY. Brute force tools show libssh, paramiko, or Go. A fourth signal is source IP: legitimate users come from known addresses, brute force comes from VPS/hosting provider IP ranges or known scanner infrastructure — verifiable via ASN lookup."
What's the difference between a Wireshark display filter and a capture filter? +
Strong answer: "Capture filters use BPF (Berkeley Packet Filter) syntax and run during packet capture — they determine what gets recorded. For example, port 22 in tcpdump. These are efficient because they reduce file size, but you can't recover packets you didn't capture. Display filters use Wireshark's own syntax and apply to an existing pcap — they show or hide packets without removing them. tcp.port == 22 is a display filter. For forensic analysis I prefer capturing broadly then filtering in Wireshark, because you preserve evidence you might not know you need yet. For long-running production captures where storage matters, targeted capture filters make sense."
How would you use packet data to validate that a security control like fail2ban is actually working? +
Strong answer: "I'd take a before/after approach. Before hardening: capture SSH traffic and document the baseline — SYN rate, unique attacking IPs, completed SSH handshakes. After applying fail2ban: capture again under identical conditions. For banned IPs I should see TCP connections completing — SYN, SYN-ACK, ACK — but then an immediate RST from the VM with no SSH version exchange, because iptables is rejecting the connection before sshd can respond. The critical metric is: do any SSH handshakes complete from previously banned IPs? If I see version strings from a banned IP, fail2ban isn't working. I'd also compare Sentinel syslog data — failed auth event count should drop to near zero for banned IPs. Packet evidence and log evidence together give you definitive validation."

SOC Context & Scenario Questions

A Tier 1 analyst escalates a Sentinel alert: "SSH brute force detected." As Tier 2, what's your investigation workflow? +
Strong answer: "I'd start by reviewing the Sentinel incident — look at the KQL output for volume (how many attempts, over what time), the attacker IPs, and whether any auth attempts succeeded. If the environment has network packet capture enabled (traffic analytics, NSG flow logs, or Azure Network Watcher), I'd pull a pcap or flow log for that IP during the alert window. In the packet data I'm looking for: tool fingerprinting via SSH version strings (for threat intel), whether the attack is still active, and whether any connections lasted longer than the brute force average (potential successful auth). I'd cross-reference attacker IPs against threat intel platforms — VirusTotal, AbuseIPDB, Shodan. Then document the evidence chain: network IOCs from pcap, auth events from syslog, alert threshold breach from Sentinel. Escalate to Tier 3 if any connections show successful auth or command execution patterns."
What is T1110 in MITRE ATT&CK and what network indicators would you look for? +
Strong answer: "T1110 is the Brute Force technique under Credential Access. It has four sub-techniques: .001 Password Guessing (many passwords against one account), .002 Password Cracking (offline), .003 Password Spraying (one password against many accounts), and .004 Credential Stuffing (using breached credential pairs). For network detection, I look for: high SYN rate to the service port (SSH :22, RDP :3389, SMTP :25), short connection duration distribution (all sessions under 2 seconds), non-standard SSH client version strings for .001/.004, and a wide IP distribution with slower per-IP rate for .003 (spraying tries to stay under detection thresholds). In Wireshark, the IO graph spike pattern also differs — .001 shows a flat high rate from one IP, .003 shows distributed lower rates from many IPs summing to the same total."
You're building a SOC playbook for SSH brute force. What network-level detection criteria would you include? +
Strong answer: "My network-level detection criteria: Threshold triggers — >10 TCP SYN packets/minute to port 22 from a single external IP; average SSH session duration <3 seconds for more than 5 consecutive sessions from same IP; RST rate >80% of SYN rate to port 22. Signature indicators — SSH client version strings matching known attack tools (libssh, paramiko, SSH-2.0-Go); KEX algorithm lists with fewer than 5 entries or missing modern algorithms like curve25519. Behavioral indicators — Sub-100ms timing between connection attempts; sequential ephemeral source port usage; connections active during off-hours from unexpected geographies. I'd automate detection via NSG flow logs feeding Sentinel, and complement with tshark-based scripts for periodic pcap analysis if the environment supports it."
